## 1. Описание реализации

В рамках данного практического занятия была выполнена миграция серверной части приложения с PostgreSQL на документоориентированную СУБД **MongoDB**. Главной целью было продемонстрировать гибкость архитектуры, построенной на принципах **SOLID**, где замена источника данных не требует переписывания бизнес-логики.

### Ключевые изменения:

1.  **Внедрение MongoDB**:
    *   Данные хранятся в виде BSON-документов в коллекциях `users`, `products`, `refreshtokens`.
    *   Используется библиотека **Mongoose** для моделирования данных (ODM) и валидации схем.
    *   Отказ от жесткой схемы SQL в пользу гибкой структуры документов, что упрощает масштабирование.

2.  **Архитектурные преимущества (SOLID & OOP)**:
    *   **DIP (Dependency Inversion Principle)**: Контроллеры по-прежнему зависят от абстракции `IRepository`. Замена PostgreSQL-репозиториев на MongoDB-репозитории прошла бесшовно для остальной части системы.
    *   **SRP (Single Responsibility Principle)**: Логика маппинга данных (преобразование `_id` в `id`, `snake_case` в `camelCase`) инкапсулирована внутри репозиториев.
    *   **Инкапсуляция**: Сложные запросы к базе данных скрыты за простыми методами интерфейса (`findAll`, `create`, `update`).

3.  **Особенности реализации**:
    *   **Асинхронность**: Полное использование `async/await` для работы с драйвером MongoDB.
    *   **Валидация**: Использование встроенных валидаторов Mongoose (`required`, `unique`, `enum`) на уровне моделей.
    *   **Безопасность**: Хеширование паролей (`bcrypt`) и JWT-аутентификация остались неизменными, так как находятся на уровне сервиса, а не базы данных.

## 3. Инструкция по запуску

### Предварительные требования
1.  Установленный **Node.js** (v16+).
2.  Установленная и запущенная **MongoDB** (локально или облачный кластер Atlas).
    *   *Для Arch Linux:* `mongod --dbpath ~/mongo-data --port 27017`
3.  Менеджер пакетов `npm`.

### Шаг 1: Настройка Backend
1.  Перейдите в папку `backend`:
    ```bash
    cd backend
    ```
2.  Установите зависимости:
    ```bash
    npm install
    ```
3.  Создайте файл `.env` и укажите строку подключения:
    ```env
    PORT=3000
    CLIENT_URL=http://localhost:3001
    ACCESS_SECRET=your_access_secret
    REFRESH_SECRET=your_refresh_secret
    
    # Локальная MongoDB
    MONGODB_URI=mongodb://127.0.0.1:27017/tea_shop_db
    
    # Или MongoDB Atlas (пример)
    # MONGODB_URI=mongodb+srv://user:pass@cluster.mongodb.net/tea_shop_db
    ```
4.  **(Опционально) Создайте администратора**:
    Если база данных пуста, выполните один раз:
    ```bash
    node src/config/initAdmin.js
    ```
5.  Запустите сервер:
    ```bash
    node server.js
    ```

### Шаг 2: Настройка Frontend
1.  В новом терминале перейдите в папку `frontend`:
    ```bash
    cd frontend
    ```
2.  Запустите клиентское приложение:
    ```bash
    npm start
    ```
3.  Приложение доступно по адресу `http://localhost:3001`.

## 5. Сравнение с Практикой №19 (PostgreSQL)

| Характеристика | Практика 19 (PostgreSQL) | Практика 20 (MongoDB) |
| :--- | :--- | :--- |
| **Модель данных** | Реляционная (Таблицы, Строки) | Документная (Коллекции, JSON-документы) |
| **Схема** | Жесткая (SQL DDL) | Гибкая (Mongoose Schema, опционально строгая) |
| **Связи** | JOIN, Foreign Keys | Ссылки (`ref`) или встраивание |
| **ORM/ODM** | Sequelize / pg | Mongoose |
| **Идентификаторы** | SERIAL / UUID | ObjectId (автоматически) |
| **Преимущество** | Целостность данных, транзакции ACID | Гибкость разработки, скорость чтения/записи |
